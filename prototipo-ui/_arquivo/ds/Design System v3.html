Por que Linear × Attio

O ERP é usado por Larissa num balcão 1280×1024 e por Wagner num escritório 1440px. Precisamos densidade e credibilidade.

Linear traz

Densidade do operador. Sidebar escura, hover states sutis, microtipo mono, espaçamentos justos. Não tem celebração visual — tem produtividade.

Attio traz

Credibilidade do dado. Off-white warm, cores-por-significado (verde dinheiro · amber atraso · azul info), tabelas com ar, ageing visual.

Os 5 princípios

Vêm da memória KB-9.75. Aterrados nas decisões deste DS.

P1 — Densidade balcão

1280×1024 é o monitor canônico. Tudo deve caber sem rolagem horizontal. Cards e gaps justos.

P2 — Atalhos visíveis

Toda ação primária tem tecla, e a tecla é mostrada. ⌘N · ⌘K · J/K · Esc.

P3 — Contexto operacional

Cada item carrega vínculo · autor · frescor. Cliente vira card, #OS vira link, NF vira preview.

P4 — IA como copiloto

IA propõe, humano decide. Sempre cita fonte. window.claude com edição revisável.

P5 — Saída mundo real

Tem que sair da tela: imprimir com brand, modo apresentação, exportar CSV/PDF, colar ao lado da Roland.

2

Proibições explícitas

Tudo abaixo já apareceu no projeto e causou ruído. Não repetir.

Não fazer

  • Inventar cor fora dos tokens · neutrals, accent, origins, semantic — fim. Use oklch() só se for um origin novo aprovado.
  • border-radius ≥ 12px em componentes operacionais (botões, badges, tabela, cards). Só drawer/modal.
  • Sombra forte em cards estáticos. Borders bastam.
  • Gradiente em background de tela ou card. Linear não tem; nós também não.
  • Emoji em UI cliente-facing. Use ícones SVG.
  • Modal full-screen pra detalhe. Use drawer lateral (PT-02).
  • Inglês em UI cliente-facing. "Vendas" não "Sells"; "Cliente" não "Customer".
  • CTA WhatsApp cliente-facing dentro do produto.
  • Redeclarar tokens em CSS de módulo. Token tem 1 lugar: tokens.css.
  • Redeclarar componentes com mesma classe. Se uma classe já existe no DS, reuse.
  • Inventar nome de classe novo antes de checar o DS. Se você precisa de algo novo: atualize o DS, depois aplique.

C3

Container queries

Layout responde ao container, não ao viewport. Crucial quando o drawer abre e reduz a área útil.

Como usar

Marque o shell com .cq-host (vira container-type: inline-size). Colunas opcionais ganham .cq-col-optional (somem <720px do container). Use .cq-hide-sm e .cq-stack-xs pra elementos que devem colapsar quando o painel encolhe — independente do tamanho da janela.

<main class="cq-host">            // container-type: inline-size
  <table class="datatable">
    <td class="cq-col-optional">…</td> // some <720px do container
@container panel (max-width: 720px) { .cq-col-optional { display: none } }

Tokens semânticos

Camada de intenção sobre os primitivos. Componentes consomem estes — re-tematizar é mexer só aqui.

Token semânticoAponta paraUso
--color-action-primary--accentfundo de botão primário, links de ação
--color-text-danger--danger-fgerro de validação, valores negativos
--surface-raised--surfacedrawer, popover, card flutuante
--border-focus--accentcontorno de campo focado
--focus-ring0 0 0 3px softhalo de foco (a11y)

Por que importa

Hoje um botão usa var(--accent) direto. Se amanhã o "primary" virar verde só pra um tenant, você troca 1 token semântico em vez de caçar --accent em 40 componentes. É a diferença entre re-tematizável e refém da cor.

E2 · decisão

Radix vs. próprio

A grande escolha: de onde vem o comportamento + a11y dos componentes no repo React.

✓ Recomendado — Radix Primitives headless

Adotar @radix-ui/react-* como base sem estilo e pintar com nossos tokens. Ganha de graça: foco gerenciado, navegação por teclado, ARIA correto, posicionamento (Popper), portal, dismiss. Cobre: Popover, Dropdown, Combobox (via Ark/cmdk), Dialog, Tabs, Switch, Tooltip, Select, Date (via react-day-picker).

✗ Evitar — reescrever do zero

Reimplementar foco-trap, roving-tabindex e posicionamento é ~3 semanas + risco alto de bug de a11y. Esses são problemas resolvidos. Gaste engenharia própria no domínio (FSM de OS, frescor, origins, ⌘K cross-module) — não em reinventar um combobox acessível.

// Padrão: Radix headless + classe do nosso DS
import * as Popover from '@radix-ui/react-popover';

<Popover.Root>
  <Popover.Trigger className="btn ghost">Ações</Popover.Trigger>
  <Popover.Portal>
    <Popover.Content className="popover">      // ← nossa classe, comportamento do Radix
      <button className="menu-item">Avançar</button>
    </Popover.Content>
  </Popover.Portal>
</Popover.Root>

// Resultado: a11y + teclado + posicionamento do Radix, visual 100% nosso.
Mapa Popover/Dropdown/Dialog/Tabs/Switch/Tooltip/Select → Radix · Combobox+⌘K → cmdk (Vercel) · Date → react-day-picker · Data viz → próprio (já é nosso diferencial)

E4

Visual regression (setup)

O que impede uma mudança de CSS quebrar 10 telas em silêncio.

Setup recomendado pro repo

Playwright + snapshot (ou Pest 4 browser). Uma página de stories renderiza cada componente em 3 densidades × 2 temas. CI roda playwright test --update-snapshots em PR; diff visual não-aprovado bloqueia o merge. 43 componentes ≈ 258 snapshots. Roda em ~40s.

// stories.spec.ts — um teste cobre toda a matriz
for (const theme of ['light', 'dark'])
  for (const density of ['compact', 'default', 'comfy'])
    test(`ds ${theme}/${density}`, async ({ page }) => {
      await page.goto(`/ds-stories?theme=${theme}&density=${density}`);
      await expect(page).toHaveScreenshot();   // falha se diferiu do baseline
    });
Gate CI bloqueia merge se o diff visual não foi explicitamente aprovado. Fim de "mudei o CSS e não sei o que quebrou".

Contrato com o Code

Regras explícitas que evitam os conflitos de CSS atuais. Detalhado em CODE_DESIGN_CONTRACT.md.

Regra única

Se a feature pediu algo que não está no Design System, o Code PARA, atualiza o DS primeiro, e só depois aplica. Sem isso, é gambiarra garantida.

Checklist pre-commit

Antes de cada commit

  • Cor: usei só tokens do DS? (zero hex/rgb/oklch hardcoded fora dos tokens)
  • Componente: classe existe no DS? Se não — atualizei o DS primeiro?
  • Spacing: usei --s-N? (zero padding: 17px mágico)
  • Radius: usei --radius*? (zero border-radius: 14px inventado)
  • Type: escala canônica? (zero font-size: 13.7px ad-hoc)
  • Sem redeclaração: a classe não está duplicada em outro arquivo CSS?
  • Sem CSS novo desnecessário: tudo que escrevi tem razão de ser?

Plano de migração — do caos atual

Hoje temos 7 arquivos CSS no Cowork. Plano em 4 ondas.

Onda 1 · Extrair tokens

Criar tokens.css no repo com os tokens desta página. Remover redeclarações de tokens em vendas.css, financeiro.css, compras-page.css, etc.

Onda 2 · Consolidar componentes

Criar design-system.css com as classes canônicas desta página (.btn, .badge, .pageheader, .moduletopnav, .datatable, .fsm-stepper, etc.). Auditar e deletar duplicatas.

Onda 3 · Migrar módulos

Por módulo (1 por sprint): trocar classes os-*, vd-*, fin-* antigas pelas canônicas. Apagar CSS legacy do módulo.

Onda 4 · Visual regression

Snapshots por tela em compact/default/comfy × light/dark. CI bloqueia regressão. Aceitar plateau.

Resultado esperado

De 7 arquivos CSS, 1.500+ classes e ~6.000 linhas → 2 arquivos (tokens.css + design-system.css) + ~800 linhas. Zero conflito porque tudo aponta pro mesmo lugar.
