1.1

5 princípios não-negociáveis

Eles guiam toda decisão. Quando há trade-off, ganha o princípio mais alto.

P1

Densidade do balcão

Larissa em monitor 1280×1024 precisa ver tudo sem rolar. Cada pixel justifica.

Como prova: tri-pane, controle de densidade global, nada de rounded-xl.
P2

Atalhos visíveis

Operador veterano não usa mouse. Toda ação primária tem tecla, e a tecla é mostrada.

Como prova: ⌘K · / · J/K · N · B · A · Esc — listados na sidebar.
P3

Contexto operacional

Cada artefato (artigo, conversa, OS) carrega o contexto ERP — vínculos, autor, frescor.

Como prova: "Anexar a OS" nativo, pílulas de frescor, contador de leituras.
P4

IA como copiloto, não autor

IA propõe, humano decide. Sempre cita fonte, sempre permite editar antes de aplicar.

Como prova: Resumir · Perguntar com [a3] · Auto-sugest revisável.
P5

Saída no mundo real

Conhecimento precisa sair da tela — imprimir, exportar, colar ao lado do equipamento.

Como prova: Imprimir SOP com header Oimpresso, modo apresentação, PDF.

7 etapas do refino — ordem que importa

Não pula etapa. Pular dá retrabalho. Cada uma fecha um gap antes de abrir a próxima.

Etapa 1 Bench

Avaliar honestamente em 15+ dimensões contra 7 referências de mercado. Score 0–10.

Etapa 2 Fundação

Layout estrutural + tokens. Tri-pane, command palette, navegação por teclado.

Etapa 3 IA básica

Resumir + perguntar com window.claude. Resposta cita fonte.

Etapa 4 Curadoria

Editor real, frescor, comentários inline, histórico de versões.

Etapa 5 Hierarquia

Subcategorias, trilhas, cross-link automático, biblioteca de fluxos.

Etapa 6 Reach

Responsive tablet, modo apresentação, favoritos, imprimir com brand.

Etapa 7 Bench v2

Re-avaliar. Aceitar plateau quando custo/benefício inflexiona.

1.4

2 pré-requisitos arquiteturais

Sem isso, features acima são gambiarra. Decidir antes de codar.

Integração cruzada (anexar & ser anexado)A1

Todo objeto do ERP (OS, cliente, artigo KB, conversa Inbox, ordem de compra) precisa aparecer no contexto dos outros. Não basta link — precisa "anexar" e "ver vínculos".

  • KB → OS: ✓ feito (botão Anexar)
  • Inbox → OS/Cliente: ✓ parcial (Linked-apps panel)
  • Inbox → KB: ✗ falta (sugerir artigo durante atendimento)
  • CRM → Inbox + KB: ✗ falta (timeline unificada do lead)
Multi-tenant + grupo econômicoA2

Wagner tem mais de 1 gráfica. Filiais compartilham dado base mas têm caixa separado. Refator estrutural — tudo lá embaixo precisa carregar tenant_id + group_id.

  • Hoje: mono-tenant — empresa única hardcoded
  • Tier 0 (mergeado PR-D): Caixa Unificada já tem isolation
  • Tier 1: KB, CRM, Clientes, Produtos seguem
  • Tier 2: Holding com filiais + DRE consolidada (caminho fiscal)

Multi-empresa + grupo econômico

Estrutura para Wagner ter holding + filiais + DRE consolidada — pré-requisito do refino #2 em diante.

3 tiers de isolation que precisam existir antes de IA

A IA do refino #2 lê dados — se a isolation não estiver pronta, vazamento entre tenants é certeza. O escopo cresce em 3 níveis. Tier 0 já está implementado (PR-D mergeado em maio/2026). Tier 1 e 2 são roadmap.

Tier 0 · feito
Single-tenant por usuário

Cada usuário pertence a uma empresa. Tudo filtra por company_id. ACL canal=fila já valida isso.

Hoje: Caixa Unificada está aqui. KB ainda não — todo dado é compartilhado.
Tier 1 · próximo
Multi-tenant — filiais

Uma conta tem N filiais. Operador pode pertencer a filial X mas ver Y. Tudo carrega company_id + branch_id.

Caso real: Larissa (ROTA LIVRE Centro) precisa ver pedido da unidade Sul para retirada cruzada.
Tier 2 · futuro
Grupo econômico — holding

3+ empresas distintas (CNPJs separados) sob uma holding. DRE consolidada na holding. Cada empresa fatura no seu CNPJ otimizado.

Caso fiscal: ROTA LIVRE Indústria (lucro real) + ROTA LIVRE Comércio (simples) — caixa de cada uma, DRE consolidada na holding.
