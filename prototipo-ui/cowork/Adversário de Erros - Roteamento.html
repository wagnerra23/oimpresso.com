Adversário de Roteamento de Erros

"Os erros chegam pelo WhatsApp? E-mail? Log? OTel?"

Você listou canos. O adversário recusa a lista e devolve a pergunta certa: qual erro, pra quem, exigindo qual ação, até quando.

Réu: a estratégia de alerta Acusação: canal sem triagem é máquina de ruído ERP: Oimpresso · ROTA LIVRE

"WhatsApp, e-mail, log, OTel" não é uma estratégia de erro — é uma lista de tubos. Tubo sem severidade, dono, deduplicação e prazo é um gerador de ruído, e ruído tem um destino só: ser silenciado. Manda todo erro pro WhatsApp e em uma semana ninguém olha. A pergunta nunca foi "por onde" — é qual erro merece interromper um humano, qual se resolve sozinho, e qual só vira estatística. E tem um erro de fundo na própria pergunta: tratar o erro do operador (Larissa) e o erro do construtor (você/os agentes) como se fossem a mesma coisa. São opostos.

Os canos, indiciados um a um

WhatsApponde a Larissa atende clientearmadilha

Misturar erro de sistema no mesmo lugar das conversas de cliente é veneno duplo: ou o erro afoga o cliente, ou o cliente afoga o erro. E um ping de WhatsApp não tem ack, escalonamento nem dedup — não é máquina de estado. Reserve pra UM caso: o cutuque humano de severidade máxima (S0), com texto de ação, não a mangueira.

E-mailcaixa de entradacemitério

Alerta por e-mail é o túmulo clássico: sem dedup, sem severidade, sem plantão. O e-mail das 2h é lido às 10h, se for. Serve pra digest (o resumo diário do que aconteceu), nunca pra "aja agora".

Logarquivo / stdoutpassivo

Log é forense, não alerta — é pull, não push. Ninguém vigia log proativamente. O erro que só existe no log é o erro que ninguém sabe até o cliente reclamar. Foi exatamente o servidor stale por 19 dias: estava "no log", mas ninguém foi empurrado. Log é onde você investiga depois que o alerta tocou — não a fonte do alerta.

OTeljá no código (OtelHelper)encanamento

OTel é poderoso, mas é encanamento, não destino. Um span não é um alerta. Sem um collector rodando + regras de alerta + dashboards, é o pecado do adversário anterior: instrumentação sem plateia. OTel vira valor quando uma regra (erro-rate, burn de orçamento, latência) dispara → e aí roteia pro canal certo. Ele alimenta a decisão; não é a decisão.

O erro de fundo: dois públicos, tratamento oposto

Erro do operador

Larissa no balcão · cliente da gráfica
  • Nunca vê stack trace, span ou código de erro.
  • Vê mensagem humana + caminho de recuperação: "Não consegui emitir a NF-e — tente de novo / salvar rascunho / chamar suporte".
  • O erro técnico vai pro público construtor em silêncio, por baixo.
  • Bloqueou dinheiro/fiscal? Vira S0 também pro lado construtor.
  • Canal = a própria UI, no fluxo dela. Não WhatsApp, não e-mail.

Erro do construtor

Você + os agentes · quem mantém o ERP
  • Recebe o técnico cru: exceção, trace, span OTel, query.
  • Roteado por severidade + dono, deduplicado e agrupado.
  • Vira ticket / ADR / sessão — entra na sua memória institucional.
  • Os gates e a sentinela são consumidores automáticos (drift, regressão).
  • Só o S0 interrompe humano; o resto é dashboard + digest.

Como cada erro procede até a resolução (o pipeline)

detecta→ classifica→ deduplica→ roteia→ ack / escala→ resolve→ aprende
  1. Detecta — handler global de exceção + OTel + health checks. Toda falha nasce capturada, não "no log por acaso".
  2. Classifica — atribui severidade (S0–S3) + público (operador/construtor) + dono. Sem isto, nada roteia direito.
  3. Deduplica/agrupa — 1000 ocorrências do mesmo erro = 1 alerta, com contador. Sem agrupar, qualquer canal se mata sozinho.
  4. Roteia — pela régua acima: S0 cutuca 1 humano; S1 vai pro ops; S2/S3 viram dashboard/digest/log. Operador recebe UX de recuperação, não trace.
  5. Ack / escala — alguém (ou um agente) confirma que pegou. Sem ack em X min → escala. Alerta sem dono é alerta morto.
  6. Resolve — corrige; S0/S1 viram incidente com timeline. Auto-curáveis (retry, restart) se resolvem sem acordar ninguém.
  7. Aprende — post-mortem vira ADR/sessão na sua memória (o loop que já existe). A sentinela/gate ganha uma regra pra aquele erro não voltar mudo. Erro sem aprendizado volta.

Veredito

Não escolha "por onde" — escolha uma régua. Um erro só procede bem se responde quatro coisas: quão grave (S0–S3), de quem é (operador ou construtor), qual a ação (auto-cura / cutuca / ticket / dashboard) e até quando (SLA). Os canos viram consequência disso: S0 → WhatsApp/push de 1 humano; S1 → ops/ticket; S2 → dashboard + digest por e-mail; S3 → log + OTel, nunca alerta. E a lei de ouro: a Larissa nunca vê um stack trace — ela vê um caminho de volta; o trace vai, calado, pro construtor. Alertar de menos custa um incidente; alertar de mais custa todos os alertas (ninguém olha). A disciplina é supressão, não amplificação.
