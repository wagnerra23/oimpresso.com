§4 A ponte git — protocolo zero-toque, sistema inteiro cole 1× no Code

! Como isto vai pro git (e o que eu NÃO faço)

Eu (Cowork) não escrevo no git — minhas tools são read-only. O protocolo é o seu já validado: eu entrego um prompt completo, você cola uma vez no Claude Code, ele cria os arquivos, roda os testes e abre um PR por onda (S1…S5), sob seu OK. Nada é commitado por mim; o Code resolve com o prompt.

S1 · ORQUESTRADOR  npm run ds:critic -- <tela>
  - Envelopa grade+report+5 guards como LENTES; agrega em 1 saída:
    governance/critiques/<tela>.critique-score.json + CRITIQUE_INDEX.md
  - NÃO reimplementa as lentes; só chama e junta. σ=0 preservado.

S2 · SEVERIDADE 0–4 + VETO-P0
  - Mapeia score(0-100)+guards → P0..P3. Regra dura: 1 P0 = reprova, ignora média.
  - Portão F1.5: passa se (0 P0) E (nota ≥ baseline). Senão → vira proposta de onda.

S3 · LOOP ESCAPE→GATE (antifrágil)
  - Comando ds:critic --learn "<descrição do escape>": registra em LICOES_CC,
    adiciona caso ao baseline certo, e ANDAIMA um check novo (stub + teste).
  - Reusa o padrão --baseline dos ratchets. Curador valida o caso.

S4 · DRIFT-WATCH + HOLDOUT
  - Liga dual-brain (v2 do README): watcher compara score vs sinal real
    (reclamação [W] / retrabalho / reabertura). Alarme se divergem.
  - governance/holdout.json: pool ROTATIVO de telas que o gate não treina.
    [W] aplica olho nelas. Passar gate + falhar holdout = flag de Goodhart.

S5 · META-EVAL + TRAVA ÚNICA
  - ds:critic --meta: taxa de escape (cai?), auto-consistência (mesma tela 2× = mesmo?).
  - 1 job CI consolida TODOS os --check (cataraca única). Regressão barra o build.

GOVERNANÇA / SOBERANIA
- 1 PR por onda (S1..S5), draft, NÃO mergeia sem "vai" de [W].
- Mudança de baseline/golden = Tier-0: revisão de [W].

Dois trilhos, não um. O sistema que aprende (este doc) é 5 ondas de costura — independente das ondas de porte de tela. Eles correm em paralelo: o ds:critic pronto mede cada porte.

Por que tão poucas: 3 das 5 são costura (S1, S2, S5 só envelopam scripts que já existem); só 2 são build novo (S3 loop, S4 drift+holdout). O git já fez o trabalho pesado — você herda ~78%.

Trilho de porte de tela (separado): a fila real do ds-report.mjs tem 10 módulos (Sells, RecurringBilling, Oficina, Repair, Purchase, Admin, Whatsapp, Settings, Financeiro, Cliente) — ~10 ondas + Caixa. Esse é o outro documento (plano formal 14±2).

Resposta direta

Sistema que aprende: 5 ondas (±1) — e ~80% já está no git, então é costura + 2 builds, não construção do zero. Materialização: o prompt mestre do §4 leva S1→S5 pro git via seu protocolo zero-toque, 1 PR por onda sob seu OK. A certeza, como sempre, é o gate: o sistema está "pronto pra aprender" quando o ds:critic --meta mostra taxa de escape caindo e a trava única no CI fica verde.
