Por que as ondas
não chegam

Você viu certo: parte aplicou (o acento colorido nas linhas) e o hero continua a caixa preta. Não é aleatório — é o sintoma de um pipeline com um elo quebrado. Fui ao repo real (@main) e achei a causa. Aqui está, com a prova.

A causa, em uma frase

A produção do Financeiro não lê os meus arquivos de protótipo. Ela lê uma cópia congelada deles — cowork-canon-financeiro-bundle.css, 9054 linhas, tirada em 20/05/2026. Minhas ondas editam o protótipo vivo; o bundle de produção é uma foto antiga que só muda quando alguém a re-tira inteira. O hero preto é dessa foto.

02 — A PROVA

Não é palpite — está escrito no repo

◆ O hero preto é uma cópia datada

fonte: resources/js/Pages/Financeiro/Unificado/Index.charter.md @main
"Bundle copy CSS 9054 LOC — cowork-canon-financeiro-bundle.css importado inteiro escopado em .fin-cowork … markup canon: … fin-stat fin-stat-hero (KPI hero warm dark hue 80)" — Ondas 12-21, 2026-05-20

Traduzindo: a produção declara o hero como "warm dark, hue 80" — a caixa escura/preta do seu print. Essa definição foi copiada em 20/05. Toda a modernização (hero claro com luz roxa) veio depois, no protótipo — e nunca foi recopiada pra dentro do bundle.

◆ Dois lugares, a mesma verdade

corroboração cruzada

(1) O charter do @main diz "hero dark hue 80". (2) O seu screenshot mostra a caixa preta. Duas fontes independentes apontam o mesmo: o bundle de produção é a foto velha, anterior à modernização.

◆ Por que o acento das linhas aplicou e o hero não

a pista do seu print

O acento colorido na borda da linha é uma regra nova e aditiva — fácil de o Code colar no bundle sem conflito. O hero exige sobrescrever uma regra que já existe (.fin-stat-hero{ background: dark }). Sobrescrita só funciona se a regra nova vencer a antiga (ordem/especificidade) — e há ainda 2 bundles Financeiro quase-duplicados + um override fin-cowork no repo, então uma definição antiga pode estar recolorindo o hero depois da minha. Resultado: aditivo passa, sobrescrita perde.

03 — OS TRÊS MODOS DE FALHA

Onde, exatamente, a mudança morre

Em ordem de probabilidade, dado o que li no repo.

MAIS PROVÁVEL

1 · O bundle não foi re-copiado da versão atual

A regra do repo é "aplicar o bundle inteiro" (feedback-cowork-bundle-aplicar-inteiro). Se o Code aplicou só trechos (o acento da linha) em vez de re-copiar o bundle completo a partir do meu financeiro.css + fin-boletos.css atuais, o hero fica na versão de 20/05. É o que o print mostra.

PROVÁVEL

2 · Minha modernização do hero mora em fin-boletos.css, não no financeiro.css

Eu pus o hero claro em fin-boletos.css (arquivo separado). Se a geração do bundle copia financeiro.css mas esquece fin-boletos.css, o hero novo nunca entra — mesmo numa regeneração "completa". Risco real e silencioso.

POSSÍVEL

3 · Deploy atrasado

Pode ser que o main já tenha o hero novo, mas o build/deploy não rodou — então a produção serve o CSS antigo. Some quando você força um deploy + Ctrl+Shift+R.

04 — COMO CONSERTAR DE VERDADE

Não "reaplicar a onda" — consertar o elo

◆ O que mandar o Code fazer

  1. Re-copiar o bundle INTEIRO, não trechos: regenerar cowork-canon-financeiro-bundle.css a partir do conjunto atual do protótipo — financeiro.css + fin-boletos.css + os JSX-CSS. (Modo 1 e 2.)
  2. Resolver os 2 bundles duplicados (cowork-financeiro-bundle × cowork-canon-financeiro-bundle) + o override fin-cowork: garantir que só um define .fin-stat-hero, senão a sobrescrita do hero perde de novo. (Modo da pista.)
  3. Verificar no DOM pós-deploy: getComputedStyle($('.fin-stat-hero')).background deve conter o linear-gradient claro, não var(--text). Se ainda for escuro → é deploy (Modo 3): forçar build + hard-reload.
  4. Travar com teste: um check que falha se .fin-stat-hero tiver background escuro/dark — pra a regressão não voltar silenciosa na próxima cópia.

A lição não é "a onda falhou" — é que entre o protótipo e a produção existe uma cópia manual, e cópia manual perde pedaço sem avisar. Enquanto o hero viver como foto de 20/05 dentro de um bundle de 9 mil linhas, cada onda vai parecer que "não aplica". Conserte a cópia, não o sintoma.
