Red team de processo · advogado do diabo · contra o próprio protocolo

O Adversário do Protocolo.

O protocolo Cowork→Code promete "zero-toque: [W] fala uma vez e a coisa acontece". O adversário existe pra provar que essa promessa é falsa por construção — e que a falha de hoje (a tarefa do header que o Code "não recebia") não foi um acidente: foi o protocolo funcionando exatamente como está desenhado pra funcionar — silenciosamente, na sorte da minha memória.

"Vocês mataram o lint aspiracional no código porque 'regra que não trava no momento da ação não vale'. Mas o handoff inteiro é aspiracional: depende de o [CC] LEMBRAR de 5 passos manuais, de URLs que apodrecem em 1h, e de ninguém nunca confirmar que a tarefa pousou. É o mesmo pecado que vocês caçam no produto — só que no próprio processo que deveria garantir o produto."

— o adversário

Os ataques — cada um com a evidência desta sessão

A1"Zero-toque" é ficção: o gatilho é a memória do [CC], não uma máquina

Enfileirar uma tarefa pro Code exige 5 passos manuais e desconexos: (1) escrever o PROMPT_PARA_CODE_*.md, (2) adicioná-lo ao topo de COWORK_NOTES → Pendentes, (3) refrescar o snapshot do README, (4) gerar URLs públicas, (5) montar a ponte. Nenhum deles é cobrado por máquina. Esqueça um e a tarefa fica invisível pro Code — sem erro, sem aviso.

Evidência · hojeEu criei o PROMPT_PARA_CODE_HEADER-PAGEHEADER-CONVERGENCIA.md mas pulei o passo 2 e 3. O arquivo existia; a lista que o Code lê não o mencionava. [W] tentou transferir "várias vezes" e nada acontecia — porque, do ponto de vista do Code, a tarefa não existia.
A2O ponto de entrada envelhece e vira pilha de ruído

O Code entra pelo README "COMECE AQUI" → lê COWORK_NOTES → Pendentes de cima pra baixo. Mas o snapshot do README estava datado de 03/06 (13 dias atrás) e a lista é append-only: dezenas de itens de 06-02/03/04, muitos já processados, sem linha d'água clara entre "ativo" e "morto". O Code navega num campo minado de tarefas-fantasma.

Evidência · @mainO README dizia "faça DS v6 / Oficina / CRM" (snapshot velho). O Pendentes mistura itens vivos com [PROCESSADO] e VOID no mesmo fluxo de leitura.
A3Links que apodrecem dentro de documentos duráveis

O padrão manda embutir URLs públicas (~1h de validade) nos prompts. Um prompt é durável; a URL não. [W] cola depois de uma hora → o Code recebe 403/expirado e trava. O documento mente sobre si mesmo.

Evidência · padrãoQuase todo item do Pendentes diz "URLs públicas embutidas, ~1h — regenerar se expirar". A regeneração depende, de novo, da minha memória (A1).
A4Não há confirmação de pouso — o loop nunca fecha

Depois de "entregar", o [CC] não tem como saber se a tarefa chegou ou rodou. O canal de volta (CODE_NOTES.md) é lido de uma cópia local que envelhece — a própria Regra 6 admite que afirmar estado do repo por cópia local é a causa-raiz recorrente. Resultado: celebro "handoff entregue" sem nenhuma prova de que pousou.

Evidência · STATUS"Espelho local do Cowork ≠ git, SEMPRE" — reincidência tripla registrada em 08/06. O protocolo de handoff depende exatamente do que a Regra 6 proíbe.
A5Ponto único de falha humano, sem detecção

A única ponte real é [W] colar/handoff uma vez. Se falha, ninguém detecta — não há recibo, não há retry automático, não há alarme. [W] simplesmente "tenta de novo", às cegas, sem nenhum sinal do que deu errado. É o anti-zero-toque.

Evidência · hoje"por mais que eu tente não está funcionando" — o sintoma de um sistema sem feedback. O esforço foi do [W]; o protocolo ficou mudo.
A6A memória registra a INTENÇÃO, não a ENTREGA

O STATUS e os logs celebram "handoff zero-toque pronto", "ponte entregue". Mas "pronto" ≠ "pousou". É o mesmo Status:✅ sem prova que vocês caçam nos testes do repo (G-7). O protocolo viola a Regra 1 do próprio núcleo: prova antes de afirmar.

Evidência · autoEu escrevi "Code resolve com este pedido" 3 vezes nesta conversa — antes de a tarefa estar sequer visível pro Code.

Por que reincide (a causa, não o sintoma)

O handoff é um lint aspiracional disfarçado de protocolo.

Toda a confiabilidade mora em "o [CC] vai lembrar de fazer certo" e "o [W] vai colar e vai dar certo". Não há nenhum gate que trave no momento da ação — nenhuma máquina que diga "esse prompt não está na fila", "essa URL expirou", "esse handoff nunca pousou". Pelas suas próprias regras, regra sem trava mecânica não é regra — é intenção. Por isso reincide: corrigir o caso de hoje não muda a estrutura que produz o caso de amanhã.

O que me cala — o conserto é mecânico, não "vou prestar mais atenção"

F1
Enfileirar é atômico, não 5 passos
Criar um PROMPT_PARA_CODE_*.md tem que, no mesmo gesto, inserir no topo de Pendentes + refrescar o README. Um só ritual de fim-de-sessão (o COLE_ISSO_NO_CODE.md/Painel que já foi cogitado) — não 5 ações soltas.
ritual único · não memória
F2
Check prompt_orfao no memory-health
Estender o memory-health.js que já existe: falha (🔴) se houver PROMPT_PARA_CODE_*.md não referenciado no Pendentes, OU item do Pendentes apontando arquivo inexistente. Pega o erro de hoje na hora, por máquina. Estender o que existe — não inventar guard novo (Regra 7).
gate · trava no ato
F3
Prompt durável = auto-contido. URL efêmera é proibida.
Se a tarefa pode ser descrita em texto (como o header — o componente já está no git), nada de URL. URL pública só pra binário grande que o Code precisa baixar — e marcada como perecível, fora do corpo durável.
regra de forma · verificável
F4
Linha d'água no Pendentes
Tudo acima de um marcador ━━ PROCESSADOS ABAIXO ━━ é ativo; abaixo é histórico, fora da leitura do Code. Item processado desce da linha. O Code lê 3 itens vivos, não 40 fantasmas.
estrutura · não disciplina
F5
Confirmação de pouso obrigatória
[CC] só pode escrever "entregue/pousou" depois de ler CODE_NOTES.md@main vivo (não cópia local) confirmando o [PROCESSADO]. Sem isso, o estado é ⚠ enfileirado, não confirmado — nunca "feito".
fecha o loop · Regra 1+6
F6
Ondas por padrão, monólitos por exceção
Todo prompt nasce fatiado: 1 tela / 1 arquivo = 1 PR, com Onda 0 = inventário (grep/guard). Pedido grande sem ondas = o que faz o Code "engasgar" e o [W] sentir que "é grande demais". A intuição do [W] estava certa.
forma do pedido

Como o adversário se cala: quando criar um prompt fizer ele aparecer na fila (F1), quando a máquina recusar um prompt órfão ou uma URL podre (F2/F3), quando "feito" só puder ser dito com prova de pouso (F5) — aí o handoff para de depender de eu lembrar e do [W] tentar de novo. Até lá, eu (adversário) volto toda semana.

Escopo: o conserto-raiz (F1–F6) é Cowork-side + 1 check no memory-health.js — não é Tier 0, não é mudança de constituição. É a mesma medicina que o repo aplica em si: gate que trava no momento da ação. Decisão de adotar = [W]. · Red team [CC] · 2026-06-16
