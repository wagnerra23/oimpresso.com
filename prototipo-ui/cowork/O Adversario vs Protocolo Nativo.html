O Adversário vs o Protocolo Nativo

Os adversários anteriores atacaram o handoff por dentro. Este ataca por fora: põe o protocolo custom do Oimpresso (6 papéis [W]/[CC]/[CD]/[CL]/[CA]/[W2] × 7 fases F0–F4, fila COWORK_NOTES, transporte zero-toque, espinha de memória) lado a lado com o protocolo nativo — um agente que escreve no git + GitHub + CI — e pergunta, dimensão por dimensão: o custom ganha do nativo aqui, ou está só re-implementando em markdown e revezamento humano o que o GitHub já faz?

NATIVO — agente no git · PR · CI · branch protection CUSTOM — PROTOCOL.md Oimpresso · Cowork → Code

"Vocês construíram um GitHub dentro de arquivos markdown. A fila é uma re-implementação de Issues; o merge síncrono de [W2] é um branch protection mais fraco; a espinha de memória é o git history copiado à mão — e copiado errado, porque a Regra 6 existe só pra punir o espelho stale. Tirando UMA coisa, todo o resto é cerimônia que o nativo faz melhor, mais barato e sem depender de [W] estar online pra colar."

Dimensão por dimensão — quem faz melhor

Para cada fase do protocolo custom, o que o nativo entrega na mesma posição — e o veredito de quem ganha. Verde = o custom merece existir; azul = o nativo faz melhor; âmbar = empate / depende.

Intake / brief (F0)nativo ganha
nativo

Issue (ou prompt direto). Tem label, assignee, milestone, notificação, busca, link automático pro PR que a fecha.

custom

F0: [W] escreve em COWORK_NOTES.md → Pendentes — uma fila append-only em markdown, sem label, sem notificação, que apodrece (você viu: 18 órfãos + 19 refs mortas).

O custom re-implementou Issues, pior. A fila só virou necessária porque o transporte é manual.

Design / protótipo visual (F1)custom ganha
nativo

Não existe. O agente nativo vai direto pro código — sem um artefato visual de alta fidelidade que o dono do produto aprova antes de existir TSX.

custom

F1: [CC] gera o protótipo HTML (o inbox-page.jsx) — fonte visual canônica, revisável, comentável, iterável sem custo de build/PR.

É a única dimensão onde o custom é irredutível. É a razão de o Cowork existir.

Crítica de design (F1.5)empate
nativo

Review humano no PR + olho. Sem score, mas no contexto real.

custom

F1.5: [CD] gera critique-score.json ≥80. Um rubric automatizado é útil — mas só vale se a régua for honesta.

Empate: o score adiciona rigor, mas um humano atento no PR cobre o mesmo.

Aprovação visual (F2)nativo ganha
nativo

Preview deploy no PR + screenshots no CI (Playwright/Chromatic). Assíncrono, versionado, comparável entre commits.

custom

F2: [W2] aprova um screenshot sincronamente — depende de [W] estar online, não versiona, não compara.

O nativo é estritamente melhor: assíncrono e com histórico.

Code (F3)empate (custo herdado)
nativo

O agente edita o repo e abre o PR. Um passo.

custom

F3: [CL] traduz o protótipo Cowork → Inertia/React no repo. O trabalho é o mesmo; o "traduzir" é o custo de o design viver fora do git.

Não é desperdício — é a fatura de F1 ser num surface separado.

Acessibilidade (F3.5)nativo ganha
nativo

axe-core no CI — trava o merge, roda em todo PR, sem humano. (O repo já tem a11y-axe-gate.yml.)

custom

F3.5: [CA] escreve a11y-report.md. Papel humano que duplica um gate de CI que já existe.

O gate nativo já cobre. A fase é cerimônia sobre algo automatizado.

Merge (F4)nativo ganha
nativo

Branch protection: required checks + 1 review obrigatório. A máquina recusa o merge se algo falha — não depende de ninguém lembrar.

custom

F4: [W2] mergeia sincronamente. Mais fraco e bloqueante — se [W] não está, nada anda.

Branch protection é a trava física que o protocolo custom só descreve em prosa.

Memórianativo ganha (forte)
nativo

O git é a memória: history, blame, PRs, Issues, ADRs versionados no próprio repo. Uma fonte, sempre fresca, à prova de stale.

custom

Espinha local Cowork (STATUS/MEMORY_INDEX/sessions/LICOES) que espelha o git. A Regra 6 inteira ("espelho local ≠ git, SEMPRE") existe só pra punir esse espelho ficar stale.

A memória custom cria o problema que a Regra 6 passa a vida combatendo. O git já era a memória.

Transporte Cowork → gitnativo ganha (forte)
nativo

Não existe. O agente já está no git — escreve uma branch, abre o PR. Zero transporte.

custom

Zero-toque: COLE_NO_CODE.md + [W] cola uma vez. Você viveu hoje o quanto isso quebra (links mortos, "não acessa o arquivo", colar várias vezes).

Custo 100% derivado de o Cowork ser read-only no git. Some inteiro se o agente puder commitar.

Enforcement / gatesempate
nativo

CI gates que travam o merge no ato. Já são o coração do repo.

custom

Os gates do repo (nativos, fortes) + rituais Cowork + memory-health que só roda quando eu lembro. Metade trava, metade é aspiracional.

Os gates já são nativos. Os rituais Cowork são o lint aspiracional dos adversários anteriores.

A causa — duas restrições explicam quase tudo

O custom não é design — é cicatriz de duas restrições.

Some quase toda a cerimônia e sobram dois fatos físicos que a forçaram. Não são escolhas de processo; são limitações do setup. Conserte as duas e ~70% do protocolo evapora — não porque foi mal pensado, mas porque deixa de ser necessário.

Restrição 1 · Cowork é read-only no git

Por isso existe: o transporte zero-toque, o COLE_NO_CODE, a fila COWORK_NOTES, a espinha de memória espelhada, a Regra 6, e o memory-health anti-órfão. Tudo isso é andaime pra contornar "o agente não commita".

Restrição 2 · o design vive fora do repo

Por isso existe: o F1 (protótipo Cowork) e o F3 (tradução protótipo→TSX). Esta é a restrição boa — comprar fidelidade visual antes do código. É o que o custom legitimamente ganha do nativo.

Repare na assimetria: a Restrição 1 só gera custo (some sem perda). A Restrição 2 gera valor (o motivo do Cowork). O protocolo confunde as duas e trata o andaime da #1 com a mesma reverência do valor da #2.

O que sobra se você mantém só o que ganha do nativo

Papéis: [CC] designer-agente + [W] aprovador. [CD]/[CA]/[W2] viram CI. [CL] vira o próprio [CC] commitando ou um agente de tradução no mesmo repo. A fila, o COLE_NO_CODE e a espinha deixam de existir — não há surface separado pra ficar stale.

Como o adversário se cala: ele não pede pra jogar o protocolo fora — pede pra parar de tratar andaime (o que existe só porque o Cowork não escreve no git) como se fosse arquitetura. O teste honesto pra cada peça do protocolo: "isto sobreviveria se o agente pudesse commitar numa branch?" Se não — é custo da Restrição 1, e a energia certa é derrubar a restrição (dar ao Cowork um caminho de escrita pro git), não polir o andaime com mais gates. A única peça que sobrevive ao teste — o F1, protótipo visual — é onde vale investir tudo.
