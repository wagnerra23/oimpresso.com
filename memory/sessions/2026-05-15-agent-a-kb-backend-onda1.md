## Decisões arquiteturais tomadas

| Problema | Solução | Razão |
|---|---|---|
| Repo-wide não tinha `BelongsToBusinessTrait` (Grep negativo) | Criar `Modules/KB/Entities/Concerns/BelongsToBusinessTrait.php` módulo-local | DRY no módulo + isolado; outros módulos podem promover pra `app/Traits/` sem ADR (refactor sem mudar contrato). Padrão atual em `Modules/ComunicacaoVisual/Entities/` inline a boilerplate no booted() de cada Model — duplica código |
| Migration `kb_decision_tree_steps` tem FK auto-referencial + FK pra `kb_decision_trees.root_step_id` (circular) | FK `root_step_id` é adicionada via `ALTER TABLE` no fim da migration §8, depois que ambas as tabelas existem | Schema canônico §7 já antecipa isso — single transaction não cobre por causa do dependency ordering das migrations |
| Bridge incremental precisa de estado por business | Criar tabela utilitária `kb_bridge_state` (extra — não está nas "11 tabelas novas" do §1) | §10 menciona explicitamente `kb_bridge_state (business_id, last_bridge_at)` mas não dá DDL. Tabela é estritamente necessária pra bridge incremental funcionar |
| `is_editable=false ⇒ body_blocks IS NULL` — MySQL CHECK em JSON é frágil | Enforcement runtime via `KbNodeObserver::saving()` lançando DomainException | §3 menciona "Eloquent observer saving event" como solução escolhida. MySQL 5.7 sem CHECK enforce — Observer é portable |
| Append-only `kb_node_versions` em V1 sem trigger MySQL | `KbNodeVersionObserver::updating/deleting` lançam DomainException | §8 explicita "V1 confia no Model Observer". Trigger fica V2 |
| Steps de decision-tree precisam linkar `*_next_step_id` mas têm ordem (FK forward refs) | 3-pass: (1) cria todos os steps com `*_fix` placeholder, (2) atualiza `*_next_step_id` por position, (3) set `root_step_id` no tree | Mantém transação atômica DB + valida invariant `KbDecisionTreeStepObserver` (yes XOR fix). Trade-off: payload JSON do POST `/decision-trees` usa `yes_next_position` em vez de `yes_next_step_id` (frontend não conhece IDs ainda) |
| Schedule bridge no Kernel: itera todos os businesses? | V1 hardcode `[1, 4]` (biz=1 Wagner + biz=4 ROTA LIVRE) | Cliente piloto definido. Quando time MCP entrar (Felipe/Maiara/Eliana/Luiz), expandir via PR — sem mudar contrato |
| Permission Spatie atual `copiloto.mcp.memory.manage` vs 7 novas chaves | V1 mantém middleware `can:copiloto.mcp.memory.manage` em todos os novos Controllers; chaves novas ficam declarativas em `permissions.php` pro PermissionRegistry/UI | §12 explicita "Dívida técnica preservada: middleware `can:copiloto.mcp.memory.manage` continua canônica até PR de rename". Rename Spatie real é PR separado (migration + roles refresh) |
| `KbDecisionTreeController::update` recria steps via replace strategy | V1 simples: delete all + re-insert via `store()` | §11 não exige edit-in-place. V2 pode otimizar com diff |
| `KbBridgeFromMcpJob` lida com docs com `business_id=NULL` (docs globais legacy) | Bridge usa `$doc->business_id ?? $this->businessId` — docs globais nascem em cada biz | `McpMemoryDocument::scopeDoBusiness` mostra que NULL = global. Bridge fan-out cria 1 kb_node por biz |
| Schedule everyFifteenMinutes via `$schedule->call()` em vez de `$schedule->job()` | `$schedule->call` itera businesses em foreach | Mais flexível pra expansão multi-biz. `$schedule->job(new KbBridgeFromMcpJob)` exigiria 1 entry por biz |

## Riscos / Pendências de validação humana

- **R-A1** Hostinger MySQL version — CHECK constraint `kb_edges.from_node_id <> to_node_id` pode ser silently ignored em MySQL <8.0.16. Observer JS-side em PHP catch via `upsertEdge` defensivo (`if ($fromId === $toId) return;`). Validar versão MySQL prod antes de mergear.
- **R-A2** Bridge job em prod com 352+ docs primeira run pode demorar — `dispatchSync` no seeder pode timeout no Install. KbBridgeFromMcpSeeder usa `dispatchSync` mas swallow Throwable; em prod, prefira dispatch async + poll de `kb_bridge_state.last_bridge_at`.
- **R-A3** `KbDecisionTreeController::update` chama `store()` re-inserindo o request — pode dar conflito de validation. V1 funciona pra payload sem mudanças massivas; V2 deve ter UpdateRequest separado.
- **R-A4** `KbNodeObserver::updating` cria snapshot mesmo se mudança não foi de campos tracked (mas otimizei pra só criar se há `isDirty(<tracked>)`). Verificar Pest cobre.
