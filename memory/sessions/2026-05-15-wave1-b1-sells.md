## Decisões técnicas

1. **Dual response via `X-Inertia` header** (não feature flag) — diferente de US-SELL-002 que usou flag. Aqui o gating é por header (canon Inertia v3) — qualquer cliente pode escolher. Reduz blast radius: zero impacto em Blade fallback. **Rationale:** mais simples + serve como base flag-able no futuro caso necessário.
2. **Defer payload pesado** — show usa Inertia::defer pra `detail` (8 with()+activities+line_taxes); edit defer `form` (sell_details join 6 tabelas); drafts/quotations defer `customers` dropdown grande; subscriptions defer `customers`. KPIs/headline ficam eager (leves, contagem agregada).
3. **DataTables AJAX preservado** — drafts/quotations/subscriptions ainda têm endpoint AJAX legacy `?ajax=1`. Page Inertia faz `fetch(urls.datatable, { Accept: application/json, X-Requested-With: XMLHttpRequest })` pra alimentar tabela cliente-side.
4. **422 JSON em guards edit()** — `canBeEdited` e `isReturnExist` retornam JSON 422 quando X-Inertia presente (substitui `back()->with(...)` Blade). Frontend trata via onError do useForm.
5. **FSM safety estrito** — `Edit.tsx` useForm NÃO inclui `current_stage_id` (anti-pattern Tier 0 explícito). Mudança de stage vai pelo `FsmActionPanel` shared (futuro Show integration).
6. **Show NÃO usa drawer SaleSheet** — full-page detail (ADR 0149 divergence). Layout 2-col 8/4 reusa pattern visual do drawer mas em página.
7. **Reuso de componentes** — KpiCard, EmptyState, AppShellV2 (não duplica).
