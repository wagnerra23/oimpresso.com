# Sessão 02 — 2026-04-18 (continuação)

**Contexto de abertura:** Eliana subiu o `Modules/PontoWr2/` (scaffold da sessão 01) para o servidor de produção `oimpresso.com` (Hostinger, `/home/u906587222/domains/oimpresso.com/public_html/`). O site começou a crashar em loop.

## Incidente — crash em produção

### Sintoma

A partir de **2026-04-18 17:12:43**, cada requisição gerava:

```
ErrorException: require(/home/u906587222/domains/oimpresso.com/public_html/Modules/PontoWr2): failed to open stream: No such file or directory
  em vendor/laravel/framework/src/Illuminate/Routing/RouteFileRegistrar.php:35
  chamado por Modules/PontoWr2/Providers/RouteServiceProvider.php linha 49
```

A stack trace mostrava que o boot do `RouteServiceProvider::mapApiRoutes()` (linha 49) chamava `Route::group(...)->group(module_path('PontoWr2', '/Routes/api.php'))` e o `module_path()` retornava só o path base do módulo — o `RouteFileRegistrar::register()` tentava então `require` um diretório, falha imediata.

### Causa-raiz

A versão do **nWidart/laravel-modules** embutida no UltimatePOS é antiga e **não aceita segundo argumento** em `module_path()`. O scaffold da sessão 01 usou a convenção "moderna" da biblioteca, que não existe nesse sistema.

### Correção emergencial

Eliana desativou o módulo (via `modules_statuses.json` ou removendo a pasta) e o site voltou imediatamente.

## Descoberta — padrão Jana

Para refatorar corretamente, olhei `Modules/Jana/` (módulo proprietário da WR2 em produção). O Jana é **o** padrão real do UltimatePOS:

- `start.php` na raiz carrega `Http/routes.php` (registrado via `module.json` → `"files": ["start.php"]`)
- **Sem** `RouteServiceProvider` separado
- **Um único** `JanaServiceProvider` com métodos `registerTranslations/Config/Views/Factories/Middleware`
- Middlewares web do UltimatePOS: `['web', 'SetSessionData', 'auth', 'language', 'timezone', 'AdminSidebarMenu', 'CheckUserLogin']`
- API com `auth:api` (Passport), não Sanctum
- Lang folder com código curto (`pt`, não `pt-BR`)

## Addendum — segunda correção (InstallController)

Depois do refactor, Eliana subiu novamente (ou o módulo tentou boot em produção) e mandou o log `laravel-2026-04-18-6aabca86.log`. O erro mudou completamente — de crash global no boot para um erro pontual numa rota específica:

```
[2026-04-18 17:40:16] live.ERROR: Class Modules\PontoWr2\Http\Controllers\InstallController does not exist
  em vendor/laravel/framework/src/Illuminate/Container/Container.php:788
  acionado quando Router tenta resolver o controller da rota /ponto/install
```

### Causa

No `Http/routes.php` (sessão 02), copiei do Jana o `Route::group` de `/ponto/install` apontando para `InstallController`, mas **não criei o controller**. A rota existe, o boot passa, mas na primeira visita a `/ponto/install` o container tenta instanciar a classe e explode.

### Correção

1. **Editado** `Modules/PontoWr2/Config/config.php` — adicionados campos usados pelo InstallController:
   - `'module_version' => '0.1'`
   - `'pid' => null` (placeholder para product ID da WR2 quando houver)

2. **Criado** `Modules/PontoWr2/Http/Controllers/InstallController.php` — clonado do Jana:
   - `$this->module_name = 'pontowr2'`
   - `$this->appVersion = config('pontowr2.module_version')`
   - `index()` — tela do instalador (requer `superadmin`, checa se já instalado)
   - `install()` — roda `module:migrate PontoWr2` + `module:publish PontoWr2` + registra `pontowr2_version` em `system`
   - `uninstall()` — remove a entrada `pontowr2_version` (tabelas ficam intactas)
   - `update()` — compara versão via `Composer\Semver\Comparator` e re-executa migrations quando subiu

### Impacto

Só a rota `/ponto/install` estava quebrada — o resto do site funcionou normalmente durante esse período (não houve novo crash global). Após subir o InstallController, `/ponto/install` deve resolver a view `install.install-module` (que é do core UltimatePOS, não do módulo).

## Addendum 2 — terceira correção (migrations com classe nomeada)

Log adicional enviado pela Eliana às 17:51:

```
Class 'CreatePontoColaboradorConfigTable' not found
  em vendor/laravel/framework/src/Illuminate/Database/Migrations/Migrator.php:448
```

### Causa

As 8 migrations do PontoWr2 foram criadas com sintaxe **anonymous class** (Laravel 9+):

```php

```

A versão do Laravel embarcada no UltimatePOS é mais antiga e o `Migrator` procura uma **classe nomeada** derivada do nome do arquivo (snake_case → StudlyCase).

Confirmação via Repair/Project:

- `Modules/Repair/Database/Migrations/..._make_repair_statuses_table.php` → `class MakeRepairStatusesTable extends Migration`
- `Modules/Project/Database/Migrations/..._create_projects_table.php` → `class CreateProjectsTable extends Migration`

### Correção

Convertidas as 8 migrations para classes nomeadas:

| Arquivo | Classe |
|---|---|
| `..._create_ponto_colaborador_config_table.php` | `CreatePontoColaboradorConfigTable` |
| `..._create_ponto_reps_table.php` | `CreatePontoRepsTable` |
| `..._create_ponto_escalas_table.php` | `CreatePontoEscalasTable` |
| `..._create_ponto_marcacoes_table.php` | `CreatePontoMarcacoesTable` |
| `..._create_ponto_intercorrencias_table.php` | `CreatePontoIntercorrenciasTable` |
| `..._create_ponto_apuracao_dia_table.php` | `CreatePontoApuracaoDiaTable` |
| `..._create_ponto_banco_horas_table.php` | `CreatePontoBancoHorasTable` |
| `..._create_ponto_importacoes_table.php` | `CreatePontoImportacoesTable` |

Também removidos os type hints `: void` dos métodos `up()`/`down()` para casar com o estilo Repair/Project (PHP antigo aceita, mas o código fica consistente com o resto do sistema).

## Addendum 3 — quarta correção (métodos Blueprint modernos)

Log das 17:58:

```
Message: Method Illuminate\Database\Schema\Blueprint::id does not exist.
```

### Causa

O `$table->id()` foi adicionado no Laravel 8. A versão antiga do UltimatePOS não conhece esse método. Outras chamadas "modernas" que fiz também quebram:

| Moderno (usei) | Antigo (correto) |
|---|---|
| `$table->id()` | `$table->increments('id')` (INT unsigned) |
| `$table->unsignedBigInteger('x')` | `$table->integer('x')->unsigned()` |
| `$table->unsignedInteger('x')` | `$table->integer('x')->unsigned()` |
| `$table->unsignedSmallInteger('x')` | `$table->smallInteger('x')->unsigned()` |
| `$table->unsignedTinyInteger('x')` | `$table->tinyInteger('x')->unsigned()` |
| `$table->foreignId('x')->constrained('t')->cascadeOnDelete()` | par explícito `integer('x')->unsigned()` + `foreign('x')->references('id')->on('t')->onDelete('cascade')` |
| `->cascadeOnDelete()` | `->onDelete('cascade')` |
| `->nullOnDelete()` | `->onDelete('set null')` |
| `$table->uuid('x')` | `$table->char('x', 36)` (mais seguro — `uuid` existe desde 5.8 mas prefiro char para não depender do driver) |

Confirmação via Repair/Project: `increments('id')` + `integer('xxx')->unsigned()` + `foreign(...)->onDelete('cascade')`. **Importante:** Business e Users no UltimatePOS têm `id` como INT unsigned (não BIGINT), então minhas FKs precisam casar — por isso `integer()->unsigned()`, não `bigInteger()`.

### Correção

Aplicada em todas as 8 migrations. Tipos consistentes: tabelas internas usam `increments` (INT unsigned) quando têm PK numérica; tabelas com imutabilidade forte (marcações, movimentos BH) usam `char(36)` como PK UUID.

### Lição para CLAUDE.md

Adicionar na lista "NÃO fazer": **métodos Blueprint modernos do Laravel 8+** (`id()`, `foreignId()->constrained()`, `unsignedXxxInteger()`, `cascadeOnDelete()`, `nullOnDelete()`). Sempre usar a API clássica: `increments/integer->unsigned/foreign->onDelete`.

## Addendum 4 — quinta correção (FK self-reference em char(36))

Log das 08:45:53:

```
SQLSTATE[HY000]: General error: 1005 Can't create table `ponto_marcacoes`
  (errno: 150 "Foreign key constraint is incorrectly formed")
  (SQL: alter table `ponto_marcacoes` add constraint
    `ponto_marcacoes_marcacao_anulada_id_foreign`
    foreign key (`marcacao_anulada_id`) references `ponto_marcacoes` (`id`))
```

**Progresso importante:** as 4 correções anteriores estavam certas — as migrations estão rodando normalmente agora. O erro é no nível do MySQL, não mais do PHP.

### Causa

FK auto-referente (marcação anulada → marcação original) em coluna `char(36)`. MySQL errno 150 nesse cenário é tipicamente divergência de charset/collation entre a PK e a coluna FK no momento em que o Laravel emite o ALTER TABLE após o CREATE TABLE. `char(36)` herda o charset da base, que pode diferir sutilmente do charset do schema.

### Correção

Removida a FK auto-referente da migration 000004. No lugar:
1. Adicionado índice `marcacao_anulada_id` (permite JOIN reverso eficiente)
2. Comentário explicando que a integridade fica em:
   - **Camada de aplicação**: `MarcacaoService::anular()` valida que o ID referenciado existe antes de inserir
   - **Triggers MySQL**: já bloqueiam UPDATE/DELETE na tabela inteira, tornando a FK auxiliar

### Decisão arquitetural

Vale registrar em ADR separada? Talvez não — é uma adaptação pequena e o ADR 0008 (triggers de imutabilidade) já cobre a filosofia "defense-in-depth app + DB". Registrado aqui e em preferences.

### Outras linhas do log (ignoradas)

As 3 primeiras linhas (01:00:05, 01:00:10, 01:00:17) são do WooCommerce plugin (HTTP client e session store) — nada a ver com PontoWr2.

## Addendum 6 — Limite de 64 chars para identificadores MySQL (2026-04-19)

**Erro (após Eliana dar `DROP TABLE` em tudo e rerodar `module:migrate PontoWr2`):**

```
[2026-04-19 09:06:57] live.EMERGENCY:
SQLSTATE[42000]: Syntax error or access violation: 1059
Identifier name 'ponto_banco_horas_movimentos_colaborador_config_id_data_referencia_index'
is too long
```

**Causa:** MySQL limita qualquer identificador (tabela, coluna, índice, FK, trigger) a **64 caracteres**. O Laravel gera automaticamente nomes de índice no formato `{tabela}_{col1}_{col2}_index`. Nesse caso:

- Tabela: `ponto_banco_horas_movimentos` (29)
- Colunas: `colaborador_config_id`, `data_referencia` (21 + 15 = 36)
- Sufixo: `_index` (6)
- Total com separadores: **72 chars** — estoura em 8.

**Correção aplicada** em `2026_04_18_000007_create_ponto_banco_horas_table.php` linha 50:

```php
// antes
$table->index(['colaborador_config_id', 'data_referencia']);

// depois
$table->index(['colaborador_config_id', 'data_referencia'], 'ponto_bh_mov_colab_data_idx');
```

Nome explícito de 27 chars, bem abaixo do limite.

**Verificação defensiva:** rodei script Python analisando todos os 8 migrations — todos os demais índices/uniques/FKs auto-gerados ficam abaixo de 64 chars. O maior remanescente é `ponto_marcacoes_business_id_colaborador_config_id_momento_index` com 63 chars (passa por folga de 1). Qualquer adição futura de coluna a esse índice estoura — se isso acontecer, nomear explicitamente.

**Regra geral incorporada:** sempre que índice composto ou FK tiver nome de tabela longo (≥ 20 chars) combinado com ≥ 2 colunas, passar nome explícito no segundo argumento. Vale também para `unique()` e `foreign(...)->name()`.
