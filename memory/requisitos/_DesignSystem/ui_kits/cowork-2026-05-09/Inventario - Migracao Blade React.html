01 Contexto e estado da migração

O Oimpresso ERP é uma instalação UltimatePOS v6 estendida com 23 módulos nWidart, alguns deles fork do core, outros próprios da WR2/Oimpresso. A camada de UI vive em dois mundos coexistentes:

Camada legada
Blade + AdminLTE ~460 views · ~6.4k chamadas Form:: via shim. Renderiza via resources/views/layouts/app.blade.php e o partial partials/sidebar.blade.php que injeta o menu via Menu::render('admin-sidebar-menu').
Camada nova
Inertia v3 + React 19 + TW4 Layout único resources/js/Layouts/AppShell.tsx. Usa shadcn/ui (Radix), Lucide icons, sonner, react-hook-form, zod. Páginas em resources/js/Pages/{Modulo}/.
Bridge

02 Padrão MWART — o contrato a preservar

"MWART" no jargão deste projeto = Módulo · Widget · Ação · Rota · Tradução: a forma como cada módulo nWidart se anuncia para o shell. É o que faz Project, Repair, Jana, Manufacturing, NfeBrasil etc. aparecerem na sidebar com seus sub-itens, badges e permissões Spatie já filtradas, sem que o shell saiba quem é quem.

Princípio de migração

O shell (Blade ou React) não conhece módulos individualmente. Recebe uma árvore MenuItem[] ordenada pelo backend (com label, icon, href, children, badge, inertia, permission) e renderiza accordion. Migrar uma tela do Blade para React não pode mexer na ordem nem nos nomes do menu — só troca o destino do href.

Campo MenuItemOrigemFunção
labelModules/{Mod}/Resources/lang/pt/menu.phpTexto exibido (PT-BR obrigatório)
iconString Lucide (ex: Wrench, Package)Resolvido pelo <Icon name=…> no React; classe FontAwesome no Blade
hrefRota nomeada LaravelPonto de bifurcação Blade vs Inertia
inertiabooleanChave da migração: true = navega via Inertia; false = full page reload (Blade)
childrenArray recursivoSub-itens do módulo — ordem preservada
badgeNumber opcionalNotificações (ex: 7 mensagens não lidas)
permissionSpatie roles/permissionsFiltrado no backend antes de chegar no front

Não faça

Não duplique a árvore de menu no React. Não codifique módulos no shell. Toda mudança de menu vive no backend (menu.php de cada módulo) e os dois renderers se atualizam juntos.

03 Sidebar Blade vs Sidebar React

BLADE partials/sidebar.blade.php

  • Largura fixa tw-w-64
  • Topo: brand business + dot online verde
  • Menu via Menu::render('admin-sidebar-menu', 'adminltecustom')
  • Sem rodapé de usuário (vive no header global)
  • Cor configurável por session('business.theme_color')
  • Sem dark mode
resources/views/layouts/partials/sidebar.blade.php · 1058 B

REACT Layouts/AppShell.tsx

  • Largura w-64 (paridade) · Sheet drawer no mobile
  • Topo: brand business + dot online verde (paridade)
  • Menu accordion recursivo (MenuList → MenuEntry)
  • Auto-expande módulo da URL atual
  • Rodapé: avatar + nome + email + ThemeToggle + Sair
  • Dark mode via ThemeToggle · Toasts via Sonner
  • ModuleTopNav opcional (sub-itens horizontais)
resources/js/Layouts/AppShell.tsx · 15.8 KB

Veredicto: O AppShell já é a versão evoluída e correta. Não recriar. Toda nova tela React deve usar <AppShell> como wrapper. O protótipo de chat que entreguei antes precisa ser refeito sobre o AppShell para preservar MWART (próxima entrega, depois desse inventário).

06 Componentes compartilhados

O que já existe em resources/js/Components/shared/ — reutilizar isso primeiro, criar novo só se nada cobrir o caso.

  • PageHeader · padrão de cabeçalho de página com título + ações
  • PageFilters · filtros padronizados
  • DataTable · tabela base com sort/paginação (TanStack Table)
  • BulkActionBar · barra de ações em lote
  • EmptyState · estados vazios
  • KpiCard · KpiGrid · cartões de KPI dashboards
  • ModuleTopNav · barra horizontal sob o header com sub-rotas do módulo (alimentada por menus/topnav.php)
  • StatusBadge · badges de status padronizados
  • SimpleMarkdown · render de markdown leve
  • Components/shared/ponto/ · componentes específicos do PontoWr2 (marcação, banco horas)

Layer de UI primitiva: Components/ui/* (shadcn) — Button, Sheet, DropdownMenu, Avatar, Tooltip, Dialog, Tabs, Toast, Switch, Select, etc. Tudo Radix.

07 Onde o módulo Chat se encaixa

Não há um módulo Chat hoje em modules_statuses.json. Conversas internas estão dispersas (provavelmente no Copiloto, na timeline do Project, ou em e-mail). O protótipo que entreguei antes deve virar:

Decisão recomendada

Criar Modules/Chat/ seguindo padrão Jana/Repair/Project. Telas em resources/js/Pages/Chat/, usando <AppShell>. A sidebar atual (com seletor de empresa no topo, módulos accordion no meio, usuário no rodapé) já existe — o chat fica só na área de conteúdo, NÃO recria sidebar.

O sub-menu do Chat (Caixa de entrada / Equipes / OS / Clientes / Configurações) usa <ModuleTopNav> horizontal, não coluna intermediária — assim mantém o ritmo visual dos outros módulos do ERP.

Fase 0 — Base já feita concluído

  • AppShell.tsx com sidebar accordion + dark mode + mobile drawer
  • LegacyMenuAdapter ligando o array Blade ao Inertia
  • Components/shared/* (DataTable, PageHeader, KpiCard, ModuleTopNav…)
  • Inertia v3 + React 19 + TW4 + shadcn estabilizados (ADR 0023)

Fase 1 — Refazer protótipo de Chat sobre AppShell próximo passo P1

  • Recriar Oimpresso ERP - Chat.html usando o esqueleto do AppShell (sidebar fixa de módulos)
  • Painel de conversas vira coluna interna do conteúdo, não substitui sidebar
  • Alinhar tipografia/cores com TW4 + tokens shadcn (HSL theme vars)
  • Definir Modules/Chat/menu.php com Caixa de entrada / Equipes / OS / Clientes

Fase 2 — Pilotar uma tela de OS (Officeimpresso) em React P2

  • Listagem de OS é o caso mais visitado — alto ROI
  • Usar DataTable + PageFilters + StatusBadge existentes
  • Manter rota legada como fallback até paridade visual
  • Validar com Eliana antes de descomissionar Blade

Fase 3 — Migrar telas de alta frequência P2

  • Clientes (CRM list/edit) · Orçamentos · Produtos
  • Cada tela: criar Page React, apontar inertia: true no menu, manter Blade como rota nomeada de fallback durante 1 sprint

Fase 4 — Telas operacionais de produção P3

  • Fila de impressão · Acabamento · Expedição (Officeimpresso)
  • Aproveitar para criar StatusBadge de etapa de produção (queue → print → finish → ship)

Fase 5 — Decomissionar Blade gradualmente P3

  • Telas de relatório/admin pouco usadas migram por último
  • NfeBrasil migra apenas após bateria fiscal aprovada
  • Meta: zerar Form:: calls e remover shim App\View\Helpers\Form

09 Riscos & checklist por tela migrada

Riscos conhecidos

1. Permissões Spatie — não duplicar lógica no front; sempre filtrar no controller antes de devolver props.

2. business_id scope multi-empresa — toda query nova precisa do scope (CLAUDE.md exige).

3. Marcações ponto_marcacoes e banco de horas: append-only por lei — nunca UPDATE/DELETE.

4. Deploy: histórico de quebras (3 incidentes em 5 dias). Sempre staging primeiro. Push em 6.7-bootstrap + composer install + artisan optimize:clear.

5. NfeBrasil & PontoWr2: zonas de teste obrigatório (fiscal/legal). Não migrar sem cobertura Pest.

Checklist por tela (cole no PR)

  • [ ] MWART preservado — menu.php não mexido; só inertia: true ativado para a rota
  • [ ] Layout — usa <AppShell>, não recria sidebar
  • [ ] Componentes shared — usou PageHeader / DataTable / etc. antes de criar novo
  • [ ] business_id aplicado em toda query
  • [ ] Permissões filtradas no controller (Spatie)
  • [ ] Lang — strings PT-BR, em Modules/{Mod}/Resources/lang/pt/
  • [ ] Fallback — rota Blade ainda existe (1 sprint)
  • [ ] Testes — Pest cobrindo regras de domínio
  • [ ] Dark mode — sem cores hardcoded; usa tokens TW4 (bg-card, text-foreground, etc.)
  • [ ] Mobile — testado no Sheet drawer
  • [ ] Toasts — flash success/error/info via Sonner já vem do AppShell
  • [ ] Staging — validado em oimpresso.test antes de push
