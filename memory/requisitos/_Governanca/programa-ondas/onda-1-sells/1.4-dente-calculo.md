# Onda 1.4 — Dente de cálculo (a prova real)

## Objetivo

O passo que prova que o mecanismo **pega o `num_uf` de verdade** — não teoria, um teste que
falha contra o bug e passa contra o fix. É o dente da dimensão D1 aplicado ao coração de Sells.

## Alvos (verificados 2026-07-02 — 0 teste hoje)

- `calculateInvoiceTotal($products, $tax_id, $discount, $uf_number)` — [app/Utils/ProductUtil.php:640](../../../../app/Utils/ProductUtil.php) — totalizador central, roda `num_uf`.
- Divergência `getTotalPaid()` (desconta devolução: `SUM(IF(is_return=0, amount, amount*-1))`) **≠** `getTotalAmountPaid()` (ignora: `->sum('amount')`) — [app/Utils/TransactionUtil.php:2992,4402](../../../../app/Utils/TransactionUtil.php).

## Passos

1. Escrever Pest com **property tests + golden fixtures**:
   - Property: `num_uf(num_f(x)) == x` para uma faixa de valores (pega o strip do ponto).
   - Golden: `227.90 − desconto 10.05% == 204.99605` **não** vira `~20.499.605` (o vetor do incidente).
   - Divergência: uma venda com pagamento `is_return=1` → `getTotalPaid` e `getTotalAmountPaid` devem ter definições **conscientemente** iguais ou o teste documenta a diferença e qual é a fonte de verdade.
2. Local do teste: `tests/Feature/Calculo/CalculoValorSellsTest.php` (ou `Modules/.../Tests`).
3. **Rodar no CT100** — proibições Tier 0: teste nunca local nem Hostinger.

## Critério de pronto

- Teste **red** contra uma versão de `num_uf`/`calculateInvoiceTotal` que strippa o ponto decimal.
- Teste **green** no código atual.
- A divergência de pagamento fica **documentada e defendida** (uma definição vence, teste trava).

## Verificação

```bash
tailscale ssh root@ct100-mcp \
  "docker exec -e DB_CONNECTION=mysql oimpresso-staging php artisan test --filter=CalculoValorSells"
```

## Por que este é o passo-chave

A REGRA MESTRE (dupla confirmação manual de cálculo) existe porque **não há** teste aqui. Este
dente converte o controle **manual** (que não sobrevive ao tempo) em controle **automático**.
É o maior ROI do programa inteiro.
